## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Implements the operator's decision (2026-07-12, on T-7): adopt plan-artifact-only BPMN now; engine execution stays open pending T-7 research.

WHAT: before non-trivial work begins, planning produces a .bpmn plan through the T-6-proven pipeline — deterministic bpmn-moddle codegen, constrained subset (single flat process, no pools/lanes/subprocesses), qq:evidence stamps on every element, bpmnlint gate, auto-layout, rendered PNG for operator review. The plan is reviewed as a diagram: every decision is a visible gateway, every step a task typed by actor (userTask = operator/judgment, serviceTask = mechanical). After execution, a post-hoc conformance check compares what actually happened against the plan's tasks and gateways and reports divergences (divergence is information, not automatically failure — replanning happens; unexplained divergence is the smell).

STORAGE & LINKAGE (operator, 2026-07-12): the plan diagram lives in Backlog docs — a plans-category document with the .bpmn and rendered PNG stored beside it — and is linked to its associated task(s) via their Documentation lists (--doc replaces the list; re-pass existing entries).

APPROVAL UX (operator, 2026-07-12): when the agent asks the operator to approve a plan — the closing confirmation of grilling — it pulls up the rendered diagram for the operator alongside the confirmation question.

BOUNDARY: openwiki process diagrams are generated by the openwiki-maintainer flow and are T-6's concern — a separate thing from plan artifacts.

FIT: same artifact format as T-6's wiki diagrams — one toolchain, no new runtime, thin-harness compatible. Wiki describes the current system; plans describe intended change.

DEPENDS: T-6 pipeline pieces (codegen helper, lint config + subset plugin, render chain). Smoke-test versions exist in the session scratchpad smoketest/bpmn/ (build-models.mjs shows the codegen pattern; run-pipeline.mjs the chain).
