# qq standing agent web access — architecture-first adoption research

**Owning Task:** T-125  
**Researched:** 2026-07-21  
**Overall confidence:** HIGH on the two named packages at their published tags and this machine's operational fit; MEDIUM on the ranked market alternative; LOW on comparative result quality because no independent benchmark was found or run.  
**What this settles:** Do **not** bless either current release as qq's standing web package today. Leave the in-flight run alone, then return to curl-only while treating `pi-web-access` as the leading *target* for a fixed-release re-evaluation. `@juicesharp/rpiv-web-tools` has the cleaner provider decomposition and stronger CI, but its single-provider/no-key failure mode and acknowledged DNS/redirect SSRF gap are worse for qq's actual unattended research use.

## Ranked recommendation

1. **No standing package today (curl via sanctioned `bash`) — recommended current disposition.** Do not disturb the in-flight user-scope `pi-web-access` run; after it finishes, remove the ad-hoc package from the standing set rather than converting v0.13.0 into policy. This is a quality hold, not a smallest-machinery judgment: v0.13.0 contains multiple typecheck-detectable `ReferenceError` paths, including an upstream-reported Pi crash, and has no CI/typecheck gate.
2. **`pi-web-access`, on a fixed release — leading adoption target.** Its end-to-end design best matches qq: zero-key Exa, reuse of Pi's Codex auth, actual runtime provider fallback, bounded timeouts, multi-stage fetch extraction, and materially stronger DNS/redirect SSRF handling. Promote it to #1 when a release fixes the unresolved identifiers, adds a real typecheck/CI gate, protects key-bearing config at 0600, and demonstrates responsive release handling.
3. **Trial `pi-web-search@1.3.1` + curl, not yet adopt.** This discovered alternative uses provider-native Google/OpenAI/Anthropic search and Pi model-registry auth with no extra provider key, including Codex OAuth; however, it deliberately has no cross-provider fallback and offers Gemini URL Context rather than a general fetch tool. The timebox permitted only a source screen, not a full audit or quality trial.
4. **`@juicesharp/rpiv-web-tools@1.20.0`.** Reconsider only if the operator already holds a supported key/backend *and* upstream adds DNS resolution + per-hop redirect validation (ideally provider failover too). Family consistency does not offset those gaps.

**What changes the verdict:** (a) a `pi-web-access` release with the three concrete runtime defects below fixed and typecheck/CI green; (b) an rpiv release with DNS/redirect-safe fetch, timeout/body bounds, provider fallback, plus a no-new-secret backend available to qq; or (c) the proposed corpus trial showing `pi-web-search` or another package materially outperforms the fixed candidates without security regressions.

## Deciding findings

### `pi-web-access` v0.13.0 (`7bdc30a`)

- **HIGH / observed / HIGH confidence — strong resilience architecture.** Search is not just provider-pluggable marketing: `gemini-search.ts` routes explicit providers and, in `auto`, catches runtime failures, preserves per-provider diagnostics, and walks OpenAI → Exa → Brave → Parallel → Tavily → Perplexity → Gemini ([`gemini-search.ts:120-259`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/gemini-search.ts#L120-L259)). Exa is coded always-available and uses its unauthenticated MCP route when no key exists ([`exa.ts:163-243,357-369`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/exa.ts#L163-L243)); OpenAI resolves Pi model-registry credentials and forwards dynamic headers, including Codex account auth ([`openai-search.ts:118-149,322-370`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/openai-search.ts#L118-L149)). Fetch has a genuine staged chain: direct HTTP/Readability/RSC, then Jina, Parallel, Gemini URL Context/Web, with abort propagation and errors retained ([`extract.ts:407-504`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/extract.ts#L407-L504)). This is substantially more failure-tolerant than rpiv's active-provider dispatch.
- **HIGH / observed / HIGH confidence — materially stronger SSRF posture.** `validateRemoteUrl` resolves every hostname, rejects private/reserved IPv4/IPv6 answers, and `fetchRemoteUrl` manually validates each redirect target before following it ([`ssrf-protection.ts:37-116`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/ssrf-protection.ts#L37-L116)); 14 focused SSRF tests cover private DNS, alternate loopback forms, redirects, IPv6, and narrow allow-ranges ([`test/ssrf-protection.test.mjs`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/test/ssrf-protection.test.mjs)). Residual: validation and connection do separate DNS lookups, so DNS rebinding is not eliminated by address pinning; body limits trust `Content-Length` before calling `text()`/`arrayBuffer()` ([`extract.ts:573-622`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/extract.ts#L573-L622)).
- **HIGH severity / observed / HIGH confidence — v0.13.0 is not type-safe and has live crash paths.** `openCuratorBrowser()` defines `sendCuratorFallbackUpdate` inside `try` and calls it from the sibling `catch`, where it is out of scope ([`index.ts:964-1189`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/index.ts#L964-L1189)); GitHub issues [#103](https://github.com/nicobailon/pi-web-access/issues/103), [#127](https://github.com/nicobailon/pi-web-access/issues/127), and [#126](https://github.com/nicobailon/pi-web-access/issues/126) report the resulting Pi crash, while fixes remain open. The same function's curator “add search” callback references an undefined `ctx` ([`index.ts:1079-1094`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/index.ts#L1079-L1094)). Fresh TypeScript 6.0.3 diagnostics reproduced both `TS2304` errors. The code intends to resolve workflow to `none` only when `ctx.hasUI === false`, but upstream #127 explicitly reports the crash in headless/no-browser use; do not treat child execution as a reliable mitigation. The current in-flight run is evidence only that at least one zero-config path works, not that this branch is safe.
- **MEDIUM severity / observed / HIGH confidence — another missing import breaks a supported path.** `gemini-web.ts` imports only `basename` but calls `readFileSync` during file upload ([`gemini-web.ts:1-3,280-295`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/gemini-web.ts#L280-L295)); the same fresh typecheck reports it. This affects Gemini-Web local-file/video upload, not ordinary search.
- **MEDIUM severity / observed / HIGH confidence — tests are useful but the release gate is structurally weak.** Fresh `npm test` passed all 69 tests, including fallback and SSRF cases. Yet the repository has no `.github/workflows`, no `tsconfig`, no TypeScript dev dependency, and its only script is `node --test` ([`package.json`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/package.json)). Several tests assert source-string patterns rather than executing extension branches; all 69 pass despite the unresolved names above. This is the deciding architecture/maintenance defect, not feature breadth.
- **MEDIUM severity / observed / HIGH confidence — key-bearing config is not permission-hardened.** The config writer uses plain `writeFileSync` without a mode/chmod or atomic replacement ([`index.ts:106-133`](https://github.com/nicobailon/pi-web-access/blob/v0.13.0/index.ts#L106-L133)), although README-directed config may contain provider keys. Environment variables and Pi model-registry auth reduce exposure, and Chromium-cookie access is opt-in, but a newly program-created file follows umask rather than enforcing 0600. No config file currently exists on this machine.
- **MEDIUM / observed / HIGH confidence — operational fit is uniquely good.** The installed v0.13.0 needs no new operator-held value: source routes unkeyed calls to Exa MCP and can reuse the existing `openai-codex` OAuth. It registers `web_search`, `fetch_content`, and `get_search_content`; it also ships a broad librarian Skill. This machine's user settings load it unpinned as `npm:pi-web-access`, so a standing choice should pin an accepted version rather than silently float.
- **HIGH concern / observed / HIGH confidence — maintenance has stalled on a release-blocking defect.** npm shows 19 releases from 2026-01-27 through 0.13.0 on 2026-06-25, MIT, one npm maintainer, and 130,072 downloads in the last observed month ([npm metadata](https://registry.npmjs.org/pi-web-access), [downloads](https://api.npmjs.org/downloads/point/last-month/pi-web-access)). GitHub showed 845 stars, 16 listed contributors but 50 commits from the lead versus 1–2 from each other contributor, and no commit after 2026-06-25 ([repository API](https://api.github.com/repos/nicobailon/pi-web-access), [contributors](https://api.github.com/repos/nicobailon/pi-web-access/contributors)). The browser-crash issue opened the release day and remained open on 2026-07-21 with multiple unmerged fix PRs. Popularity is not a substitute for an operative release path.

### `@juicesharp/rpiv-web-tools` v1.20.0 (`060373d`)

- **HIGH / observed / HIGH confidence — cleaner internal abstractions and release engineering.** `SearchProvider`, `FetchProvider`, `FullProvider`, and `ProviderMeta` separate capability and setup contracts ([`providers/types.ts:19-100`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-web-tools/providers/types.ts#L19-L100)); a central factory creates ten small provider classes ([`providers/factory.ts`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-web-tools/providers/factory.ts)), while metadata drives generic setup. Fresh tests passed **316/316** across seven files. Monorepo CI runs formatter/lint, `tsc --noEmit`, tests and coverage on Node 22 and 24 ([`.github/workflows/ci.yml`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/.github/workflows/ci.yml)). This is decisively cleaner than `pi-web-access`'s 2,598-line `index.ts` plus 3,399-line embedded curator page.
- **HIGH / observed / HIGH confidence — there is no provider fallback chain.** `web_search.execute` instantiates exactly one active provider and directly awaits its `search`; missing credentials, 429s, malformed JSON, network errors, and empty results do not try another configured provider ([`web-tools.ts:256-305`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-web-tools/web-tools.ts#L256-L305)). Fetch's documented “fallback” is capability dispatch—interceptor → that same provider's native fetch → generic HTML only when the provider has no fetch—not failover after a native provider error ([`web-tools.ts:339-395`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-web-tools/web-tools.ts#L339-L395)). Provider errors are labeled strings, not a retryable/auth/rate-limit taxonomy; hosted requests add no package timeout beyond an upstream abort signal.
- **HIGH severity / observed / HIGH confidence — its SSRF guard is explicitly incomplete.** The initial guard compares only literal hostnames/IPs ([`web-tools.ts:159-193`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-web-tools/web-tools.ts#L159-L193)); generic fetch then uses `redirect: "follow"` without revalidation ([`providers/fetch-helpers.ts:95-108`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-web-tools/providers/fetch-helpers.ts#L95-L108)). Upstream's own README states that public DNS resolving private or a 302 to private still reaches the target ([`README.md:270-274`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-web-tools/README.md#L270-L274)). This matters because search results and fetched content are untrusted even though qq accepts open delegate egress. The implementation also reads the full body before truncation/spill, with no byte/time bound.
- **HIGH / observed / HIGH confidence — unconfigured search fails on this machine.** Config defaults to Brave; Brave throws when `BRAVE_SEARCH_API_KEY` is absent ([`web-tools.ts:58,133-145`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-web-tools/web-tools.ts#L133-L145), [`providers/brave.ts:24-56`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-web-tools/providers/brave.ts#L24-L56)). Every hosted provider needs a key. SearXNG/Ollama avoid a hosted key only by requiring a running backend; neither is configured here, the relevant environment variables are unset, and no Ollama binary is present. Thus family consistency buys no immediate research capability and imposes a new operator-held value/service.
- **LOW / observed / HIGH confidence — secret persistence is better.** Shared `rpiv-config` writes JSON then best-effort chmods 0600 and reports save failure ([`packages/rpiv-config/config.ts:64-95`](https://github.com/juicesharp/rpiv-mono/blob/v1.20.0/packages/rpiv-config/config.ts#L64-L95)); `/web-tools` guards success notifications. Reads fail soft with a warning for malformed JSON. It is not atomic, and hard schema violations silently collapse to `{}`, but this remains stronger than `pi-web-access`.
- **MEDIUM / observed / HIGH confidence — active maintenance, extreme bus-factor concentration.** npm records 103 releases between 2026-04-13 and 1.20.0 on 2026-06-15, MIT, one npm maintainer, and 5,439 monthly downloads ([npm metadata](https://registry.npmjs.org/%40juicesharp%2Frpiv-web-tools), [downloads](https://api.npmjs.org/downloads/point/last-month/%40juicesharp%2Frpiv-web-tools)). The monorepo was pushed through 2026-07-10 and the package path has post-release fixes/features, but the contributor API attributes 1,212 commits to the lead and at most two to each other listed contributor ([repository API](https://api.github.com/repos/juicesharp/rpiv-mono), [contributors](https://api.github.com/repos/juicesharp/rpiv-mono/contributors)). Maintenance responsiveness is strong; bus factor remains approximately one. Because qq pins 1.20.0, post-release changes do not benefit it until a new version is deliberately accepted.

## Collision and child-session operation

- **HIGH / observed / HIGH confidence — installing both is a real collision, not merely “likely.”** Both source trees call `registerTool({name: "web_search"})`. Pi 0.80.10 keeps both extensions loaded, emits `Tool "web_search" conflicts with …`, and tool aggregation says “first registration per name wins” ([local installed source: `dist/core/resource-loader.js:401-407,780-813`; `dist/core/extensions/runner.js:252-262`]). Outcome depends on load order, so one package's search becomes unreachable while its other tools remain. Do not install both unfiltered. `pi-web-access` can unregister only its search via `webSearch.enabled:false`, but that does not produce a sensible qq default.
- **HIGH / observed / HIGH confidence — user packages reach ordinary children by construction.** Pi documents global settings at `~/.pi/agent/settings.json`, user npm installs under `~/.pi/agent/npm/`, and package resources loading from the global `packages` list ([Pi 0.80.10 `docs/settings.md:1-8,230-271`; `docs/packages.md:43-65`]). Installed `pi-subagents` documents that omitted `extensions` means “all normal extensions load” and omitted tools means normal tools ([`pi-subagents/README.md:785-815`](https://www.npmjs.com/package/pi-subagents)). qq's normal fresh delegates therefore receive whichever standing web package the user list loads; explicit child extension allowlists can intentionally exclude it.
- Current observed user settings already contain the pinned rpiv family (`rpiv-ask-user-question@1.20.0`, `rpiv-btw@1.20.0`, installed `rpiv-todo` 1.20.0) and unpinned `pi-web-access`; no `rpiv-web-tools` config or web-search config exists. No secrets were read.

## Other package and null-option screen

- **MEDIUM / observed / MEDIUM confidence — strongest discovered alternative is `pi-web-search@1.3.1`.** The pi.dev package page and npm tarball show a seven-file, MIT provider-native extension. It registers `web_search` plus Gemini-only `url_context`, selects current/configured Google/OpenAI/Anthropic models, and resolves auth through Pi's model registry; source supports Codex credentials ([pi.dev](https://pi.dev/packages/pi-web-search), [`src/api.ts`](https://github.com/ttttmr/pi-web-search/blob/118b3eee3f5900cef3141745109a300315630592/src/api.ts), [`src/index.ts`](https://github.com/ttttmr/pi-web-search/blob/118b3eee3f5900cef3141745109a300315630592/src/index.ts)). It explicitly errors rather than cross-provider fallback and has one dominant maintainer, 13 stars, 2,982 monthly downloads, and no version tags observed. This deserves the cheap trial, not immediate adoption.
- **LOW / gap — catalog breadth was intentionally timeboxed.** pi.dev also exposed `@ollama/pi-web-search` (requires Ollama backend/key; tiny 0.0.5 package, no linked repository) and browser-automation packages, while npm search found many young key-specific tools. They were not source-audited because the owner prioritized the named candidates. No screened package displaced the shortlist on both zero-new-secret fit and architecture evidence.
- **MEDIUM / inference / HIGH confidence — curl-only preserves fetch but loses a reliable search surface.** Delegates already have sanctioned `bash` egress, so curl can retrieve known URLs, registry metadata, GitHub API/source, redirects and headers without package code or tool collisions. It does **not** supply provider discovery/search, normalized result envelopes, citation extraction, page readability, fallback routing, context spill, or consistent abort/time/size policy; every agent must reconstruct those. Search-engine HTML scraping is brittle. Null is therefore an acceptable quality hold, not the long-term capability answer.

## Search/fetch quality evidence and cheap decisive trial

**Gap / LOW confidence:** No independent apples-to-apples comparison of these Pi extensions was found. Download/star counts are adoption signals, not relevance, citation, latency, or extraction quality. This run did not spend a provider key or disrupt the active research child, so it did not fabricate a benchmark from unequal backends.

**Cheap trial that would settle it:** after a fixed `pi-web-access` release, run 20 frozen prompts in fresh headless children against (A) its Codex/Exa auto path, (B) `pi-web-search` on Codex, and (C) rpiv only if an already-approved provider is available. Include recent package versions, niche source-code questions, date/domain filters, no-result queries, 429/timeout simulation, JS-heavy pages, PDF, redirect-to-private, and DNS-to-private fixtures. Record p50/p95 latency, success/failover, result precision@5, citation URL validity, full-page extraction completeness, bytes returned/spilled, and operator interventions. Require zero private-target connections and inspect every cited source. Ten paired search prompts plus ten fetch fixtures is enough to reject a candidate; it is not a general search-provider benchmark.

## Residual gaps and risks

- No live comparative result-quality/latency trial; no external independent benchmark located.
- The `pi-web-search` alternative received only a source screen, not the same full audit.
- Exa's unauthenticated MCP availability, privacy terms, quota and durability were not independently established; `pi-web-access` hard-codes it as always available.
- Neither named package pins validated DNS answers to the actual socket; PWA still has DNS-rebinding residual risk.
- Tool output from the public web remains untrusted and prompt-injection-capable in every option, including curl; neither package can make model-consumed content trusted.
- GitHub counts are snapshots and monorepo-wide issue/contributor signals can overstate maintenance specific to `rpiv-web-tools`.
