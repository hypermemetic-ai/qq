# Alignment governance

This Skill supplies governance only. Run the interaction through the
all-terminal plan loop: structured-question cards, the `qq-plan-loop` bridge,
the hunk review surface, and explicit approval selection. Its planning phase
fails closed, so enactment cannot precede approval.

## Role gate

Only the operator-facing accountable owner aligns genuinely new work. Spawned,
delegated, review, research, maintainer, and event-triggered Actors treat a
bounded assignment as aligned and execute inside it. A new consequential
decision or scope gap stops them and returns to the assigning or owning Actor.

## Dispositions and ledger

Dispositions do not transfer: each covers only the decision and surface it
settled. Authorization alone does not settle implementation shape. Every
consequential decision cites its Backlog decision record, approved Task, or
asked-and-answered alignment exchange; without a citation it is open. An
operator opt-out is a disposition only when recorded verbatim for that work.

Before enactment, the owning Task Description's **decision ledger** cites every
embedded consequential decision or says `none`. Use the plan loop to expose the
intended outcome, ownership boundary and non-goals, success evidence, and each
decision's disposition before explicit approval.

When a settled decision reaches beyond one Change, mint its Backlog decision
record in the Change checkout that first encodes it, riding that pull request—
never in primary `main`. Cite the asked-and-answered exchange until the record
exists, then switch the ledger to its id before Task finalization.

If work reveals a new consequential decision or crosses the approved boundary,
stop. The accountable owner re-enters alignment through the plan loop; a
non-owning Actor follows the return path above.
